# AGENTS.md — G4 Valley 2026 Dashboard

## Project Overview

Sales/acquisition dashboard for **G4 Valley 2026 (Lote 01)**, a G4 Educação launch event. It tracks revenue (faturamento), sales, leads, and ad spend against goals, split by channel.

- **Frontend:** single-file `index.html` (~2000 lines) with vanilla JS + Chart.js 4 loaded from CDN. Zero build step, zero npm dependencies.
- **Data pipeline:** `g4valley-fetch.py` (Python 3, only external dep: `requests`) queries the Databricks SQL Statements API and writes `g4valley-data.json`. It also embeds the JSON inline into `index.html` (as the `#inlineData` script tag / `INLINE_DATA`) so the dashboard works when opened via `file://`.
- **Deploy:** Vercel static hosting (`vercel.json`, no build command). Push to the repo triggers deploy.
- UI text is in **Brazilian Portuguese** — keep it that way.

## Reference

### index.html layout

- 3 pages toggled via `switchPage()`: `#page-overview-valley`, `#page-comparison`, `#page-ads-report` (nav items use `data-page`).
- `fetchData()` (~line 1200) loads `./g4valley-data.json` via fetch, falling back to `INLINE_DATA` for `file://`.
- Rendering functions: `updateKPIs()`, `updateChannelGoals()`, `updatePerfPremises()`, `updateCharts()`, `buildDailyTable()`.

### g4valley-fetch.py layout

- `run_query(sql)` — Databricks SQL Statements API helper (POST + poll).
- `fetch_*()` functions — one per query (vendas totais/por dia/por canal/por hora, marketing invest, meta/google ads vendas, leads).
- `build_json()` — assembles the final payload, writes `g4valley-data.json`, and re-embeds it into `index.html`.

### Key business rules (do not change without asking)

- Edition filter: `g4valley-1026`; Lote 01 window: `2026-07-07` → `2026-07-28`.
- Channel classification lives in SQL `CASE` expressions: `utm_source IN ('instagram','ig')` is reclassified as organic channels (`tg_social`, `alfredo`, `nardon`, `g4_social`) by `utm_medium` — NOT paid Meta Ads.
- Meta Ads = `utm_source = 'facebook'` + campaign `LIKE '%_adsfb_gtm_g4valley26_vendas_carrinhoaberto_alwayson%'`; Google Ads = campaign `LIKE '%_adsgg_gtm_g4valley26_carrinhoaberto_vendas_%'`.
- Tables: `g4_eventos_lancamentos.vw_mart_eventos_orders` (vendas), `g4_eventos_lancamentos.vw_mart_eventos_leads_pre_inscricao` (leads), `production.gold.marketing_fct` (ad spend).

## Patterns

- Keep everything in the single `index.html` — do not split into separate JS/CSS files or introduce a bundler/framework.
- New data needs = add a `fetch_*()` function in `g4valley-fetch.py`, wire it into `build_json()`, then consume the new JSON field in `index.html`. Keep the JSON shape backward-compatible where possible since a stale embedded copy may exist.
- Config via env vars with defaults (`DATABRICKS_HOST`, `DATABRICKS_TOKEN`, `DATABRICKS_WAREHOUSE_ID`); the token is required and must never be hardcoded or committed.
- When changing channel logic, update **both** `CASE` copies (`fetch_vendas_by_canal` and `fetch_vendas_by_day_canal`) — they duplicate the same expression.

## Anti-patterns

- Never commit `g4valley-data.json` (gitignored; regenerated by the script).
- Never commit a Databricks token or `.env`.
- Don't classify `instagram`/`ig` traffic as paid Meta Ads — it was explicitly reclassified as organic (see commit `38e73b5`).
- Don't add npm/package.json/build steps; the whole point of the stack is zero-build static deploy.
